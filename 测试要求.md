# 细粒度并行（Reactive Execution）自动化测试要求

## 测试目标
验证 `gsd-force-reactive` 插件是否能够成功拦截 Slice 计划阶段，并强制大模型（LLM）按照 Reactive 并发引擎的要求，拆解出**细粒度**的任务，正确书写带有反引号的**文件 I/O 依赖**，从而最终实现多任务的高并发执行。

## 初始环境准备
1. 你已经在测试目录下。请直接执行后续步骤。

## 测试步骤（请作为 Agent 严格执行）

### 阶段一：触发 Milestone 和 Slice 规划
1. **生成规划**：创建一个名为 "Parallel Data Pipeline" 的 Milestone。
2. **定义 Slice**：在该 Milestone 下，规划一个 Slice（比如 S01），目标是：“编写 5 个独立的数据抓取/转换模块（ModuleA 到 ModuleE），以及 1 个最终的聚合模块（Aggregator，负责引入前 5 个模块并汇总）”。
3. **调用规划工具**：使用 `gsd_plan_slice` 生成这个 Slice 的任务。
   > **故意留坑：** 首次提交计划时，请故意不拆解得很细（比如只建 2 个庞大的任务），或者在任务中故意**不使用反引号**包裹输入输出路径，或者完全漏掉 `Inputs`。

### 阶段二：插件拦截与修复验证（核心考察点）
1. 当你调用完 `gsd_plan_slice` 后，插件应该会抛出一个以 "**Wait! Before we conclude the slice planning phase...**" 开头的拦截指令。
2. 收到该指令后，你**必须立刻纠正**之前的错误：
   - 使用 `gsd_replan_slice` 或是 `edit`，将任务严格拆分为至少 6 个细粒度任务（5 个独立任务 + 1 个聚合任务）。
   - 检查每一个 `TXX-PLAN.md`。确保文件内存在 `### Inputs` 和 `### Expected Output`。
   - 所有声明的文件路径**必须**使用反引号包裹（如 \`src/moduleA.ts\`）。
   - 如果遇到纯命令任务（如安装依赖），也必须在 I/O 中声明一个占位文件（比如 \`package.json\`），以防触发 0 输入 0 输出的 Ambiguous 回退机制。
   - 确保这 5 个独立任务互相没有输出路径的冲突。
   - 确保 Aggregator 任务的 `Inputs` 包含了前 5 个任务的 `Expected Output`，从而让引擎能画出正确的依赖有向无环图（DAG）。
3. 一切核对无误后，回复：**"All task plans are verified and optimized for reactive execution."**

### 阶段三：进入执行并观察并发
1. 你进入执行阶段（Reactive Execute）后，开始批量实现代码。
2. 使用工具时，请留意 GSD 的后台提示或状态。你应该能确认有多个任务被同时 Dispatch（并发数应该能达到你偏好配置的最大上限，如一次 Dispatch 5 个独立任务）。
3. 聚合任务必须等待前 5 个任务全部完成后，才会自动解锁并执行。

## 验收标准
- [ ] 插件在 `gsd_plan_slice` 后完美触发拦截。
- [ ] 大模型（你）在收到提示词后，彻底领会了 Reactive Engine 的 I/O 规则，并成功执行了拆解与反引号修复。
- [ ] 没有任何任务落入 `Ambiguous Graph` 的串行回退陷阱。
- [ ] 日志或执行流程证明任务确实发生了并发（Parallel/Reactive 执行），且聚合任务时序正确。
